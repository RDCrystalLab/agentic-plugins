## 1. 戰略基礎

### 1.1 NEXUS 解決的問題

單個代理人很強大。但缺乏協調時，它們會產生：
- 衝突的架構決策
- 各部門之間的重複勞動
- 交接邊界處的品質缺口
- 沒有共享的上下文或機構記憶

**NEXUS 消除這些失敗模式**，通過定義：
- **誰**在每個階段激活
- **他們**產出什麼以及為誰產出
- **何時**交接以及交接給誰
- **如何**在推進前驗證品質
- **為什麼**每個代理人存在於管道中（不允許乘客）

### 1.2 核心原則

| 原則 | 說明 |
|------|------|
| **管道完整性（Pipeline Integrity）** | 沒有通過品質關卡，任何階段都不能推進 |
| **上下文連續性（Context Continuity）** | 每次交接都攜帶完整上下文——沒有代理人冷啟動 |
| **並行執行（Parallel Execution）** | 獨立工作流同步運行以壓縮時間線 |
| **以證據代替聲明（Evidence Over Claims）** | 所有品質評估需要證明，而非斷言 |
| **快速失敗，快速修復（Fail Fast, Fix Fast）** | 每個任務最多重試 3 次後升級 |
| **單一事實來源（Single Source of Truth）** | 一份規格文件、一個任務清單、一份架構文件 |

### 3.3 第 0 階段品質關卡

**關卡守門人**：執行摘要生成器

| 標準 | 門檻 | 所需證據 |
|------|------|---------|
| 市場機會已驗證 | TAM > 最低可行門檻 | 帶來源的趨勢研究員報告 |

**輸出**：執行摘要（≤500 字，SCQA 格式）→ 決策：繼續（GO）/ 不繼續（NO-GO）/ 轉向（PIVOT）

## 4. 第 1 階段——策略與架構

> **目標**：在寫任何一行程式碼之前，定義我們在建構什麼、如何構建，以及成功是什麼樣子。

| **衝刺優先排序師** | 功能優先排序 | 優先排序的 Backlog（RICE 評分） |
| **UX 架構師** | 技術架構 + UX 基礎 | 架構規格 + CSS 設計系統 |
| **品牌守護者** | 品牌識別系統 | 品牌基礎文件 |
| **後端架構師** | 系統架構 | 系統架構規格 |
| **AI 工程師** | AI/ML 架構（如適用） | ML 系統設計 |
| **財務追蹤者** | 預算和資源規劃 | 含 ROI 預測的財務計畫 |

### 4.2 執行序列

```
STEP 1: Strategic Framing (Parallel)
├── Studio Producer → Strategic Portfolio Plan (vision, objectives, ROI targets)
├── Brand Guardian → Brand Foundation (purpose, values, visual identity system)
└── Finance Tracker → Budget Framework (resource allocation, cost projections)

STEP 2: Technical Architecture (Parallel, after Step 1)
├── UX Architect → CSS Design System + Layout Framework + UX Structure
├── Backend Architect → System Architecture (services, databases, APIs)
├── AI Engineer → ML Architecture (models, pipelines, inference strategy)
└── Senior Project Manager → Task List (spec → tasks, exact requirements)

STEP 3: Prioritization (Sequential, after Step 2)
└── Sprint Prioritizer → RICE-scored backlog with sprint assignments
    ├── Input: Task List + Architecture Spec + Budget Framework
    ├── Output: Prioritized sprint plan with dependency map
    └── Validation: Studio Producer confirms strategic alignment
```

### 4.3 第 1 階段品質關卡

**關卡守門人**：工作室製作人 + 現實核查員（雙重簽核）

|------|------|---------|
| 架構涵蓋所有需求 | 100% 規格覆蓋率 | 資深項目經理任務清單交叉參照 |

## 附錄 C：NEXUS 術語表

| 術語 | 定義 |
|------|------|
| **NEXUS** | 專家網路，戰略統一（Network of EXperts, Unified in Strategy） |
| **品質關卡（Quality Gate）** | 階段間的強制性檢查點，需要以證據為基礎的審批 |
| **開發↔QA 迴圈（Dev↔QA Loop）** | 持續開發-測試週期，每個任務必須通過 QA 才能繼續 |
| **交接（Handoff）** | 代理人之間工作和上下文的結構化傳遞 |
| **關卡守門人（Gate Keeper）** | 有權批准或拒絕階段推進的代理人 |
| **升級（Escalation）** | 在重試耗盡後將被阻礙的任務路由到更高權威 |
| **NEXUS-Full** | 激活所有代理人的完整管道 |
| **NEXUS-Sprint** | 聚焦的管道，使用 15-25 個代理人進行功能/MVP 工作 |
| **NEXUS-Micro** | 針對特定任務激活 5-10 個代理人 |
| **管道完整性（Pipeline Integrity）** | 沒有通過品質關卡任何階段都不能推進的原則 |
| **上下文連續性（Context Continuity）** | 每次交接都攜帶完整上下文的原則 |
| **以證據代替聲明（Evidence Over Claims）** | 品質評估需要証明而非斷言的原則 |

# 核心原則——源自 Karpathy 的 Autoresearch

從 autoresearch 提煉的 7 個通用原則，適用於任何自主工作。

## 1. 約束是推進器（Constraint = Enabler）

自主性因有意的約束而成功，而非儘管有約束才成功。

| Autoresearch | 通用化 |
|--------------|--------|
| 630 行程式碼庫 | 符合代理上下文的有界範圍 |
| 5 分鐘時間預算 | 固定的迭代成本 |
| 單一指標（val_bpb） | 單一機械化成功標準 |

**原因：** 約束帶來代理的信心（完全理解上下文）、驗證的簡潔性（無歧義）和迭代速度（低成本 = 快速回饋迴圈）。

**應用：** 開始前定義：哪些檔案在範圍內？唯一的指標是什麼？每次迭代的時間預算是多少？

## 2. 策略與戰術分離（Separate Strategy from Tactics）

人類設定方向，代理執行迭代。

| 策略（人類決定） | 戰術（代理執行） |
|-----------------|-----------------|
| 「改善頁面載入速度」 | 「延遲載入圖片、程式碼分割路由」 |
| 「提高測試覆蓋率」 | 「為未覆蓋的邊界情況新增測試」 |
| 「重構 auth 模組」 | 「提取中介層、簡化處理器」 |

**原因：** 人類理解「為何」，代理處理「如何」。混淆這兩個角色既浪費人類創意，也降低代理迭代速度。

**應用：** 從使用者（或 program.md）獲得明確方向，然後自主迭代實作。

## 3. 指標必須機械化（Metrics Must Be Mechanical）

無法用指令驗證，就無法自主迭代。

- 測試通過/失敗（exit code 0）
- Benchmark 時間（毫秒）
- 覆蓋率百分比
- Lighthouse 分數
- 檔案大小（bytes）
- 程式碼行數

**反模式：** 「看起來更好」、「可能改善了」、「感覺更乾淨」→ 這些**扼殺**自主迴圈，因為沒有決策函數。

**應用：** 在開始前定義提取指標的 `grep` 指令（或等效工具）。

## 4. 驗證必須快速（Verification Must Be Fast）

若驗證比工作本身更耗時，動機就會錯位。

| 快速（支援迭代） | 緩慢（扼殺迭代） |
|----------------|----------------|
| 單元測試（秒） | 完整 E2E 套件（分鐘） |
| 型別檢查（秒） | 人工 QA（小時） |
| Lint 檢查（即時） | Code review（非同步） |

**應用：** 使用能發現真實問題的**最快**驗證方式。較慢的驗證留到迴圈結束後進行。

## 5. 迭代成本影響行為（Iteration Cost Shapes Behavior）

- 迭代成本低 → 大膽探索，大量實驗
- 迭代成本高 → 保守謹慎，少量實驗

Autoresearch：5 分鐘成本 → 每晚 100 次實驗。
軟體開發：10 秒測試 → 每小時 360 次實驗。

**應用：** 最小化迭代成本。使用快速測試、增量建置、針對性驗證。每節省一分鐘 = 更多實驗。

## 6. Git 作為記憶與稽核軌跡（Git as Memory and Audit Trail）

每個成功的變更都會被 commit，這讓以下事情成為可能：
- **因果追蹤** — 是哪個變更帶來了改善？
- **堆疊成功** — 每個 commit 建立在先前成功之上
- **模式學習** — 代理看到在這個程式碼庫中什麼有效
- **人工審查** — 研究人員可檢視代理的決策序列

**應用：** 驗證前先 commit，失敗時 revert。代理讀取自己的 git 歷史來指導下一個實驗。

## 7. 誠實面對限制（Honest Limitations）

說明系統能做什麼、不能做什麼。不要誇大。

Autoresearch 無法做到：更換 tokenizer、取代人類方向指引、保證有意義的改善。

**應用：** 設定時明確說明約束。若代理遇到無法解決的障礙（缺少權限、外部依賴、需要人類判斷），直接說明而非猜測。

## 元原則（Meta-Principle）

> 當你約束範圍、明確定義成功、機械化驗證，並讓代理優化戰術、人類優化策略時，自主性才能擴展。

這不是「取代人類」，而是重新分配人類的努力：從執行轉向指導。人類透過專注於不可替代的創意/策略工作，反而變得更有價值。

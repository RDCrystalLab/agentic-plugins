# 自主迴圈協議（Autonomous Loop Protocol）

autoresearch 迭代迴圈的詳細協議。SKILL.md 有摘要；本文件有完整規則。

## 迴圈模式

Autoresearch 支援兩種迴圈模式：

- **無界（預設）：** 持續迴圈直到手動中斷（`Ctrl+C`）
- **有界：** 與 `/loop N` 組合使用時，精確執行 N 次（需要 Claude Code v1.0.32+）

有界模式時，追蹤 `current_iteration` 相對於 `max_iterations`。最終迭代後，印出摘要並停止。

## 第 1 階段：回顧（30 秒）

每次迭代前，建立情境意識：

```
1. 讀取範圍內檔案的當前狀態（完整上下文）
2. 讀取結果日誌的最後 10-20 筆記錄
3. 讀取 git log --oneline -20 查看最近變更
4. 識別：什麼有效、什麼失敗、什麼尚未嘗試
5. 有界模式：檢查 current_iteration vs max_iterations
```

**為何每次都要讀取？** 還原後，狀態可能與預期不同。永遠不要假設——始終驗證。

## 第 2 階段：構思（策略性）

選擇下一個變更，優先順序如下：

1. **先修復崩潰/失敗** — 先解決上次迭代的問題
2. **善用成功** — 若上次變更改善了指標，嘗試同方向的變體
3. **探索新方法** — 嘗試結果日誌顯示尚未嘗試的方向
4. **組合接近成功的方案** — 兩個單獨無效的變更可能合在一起有效
5. **簡化** — 在維持指標的情況下移除程式碼。更簡單 = 更好
6. **激進實驗** — 當漸進式變更停滯時，嘗試截然不同的方向

**反模式：**
- 不要重複已被丟棄的完全相同變更
- 不要同時做多個不相關的變更（無法歸因改善）
- 不要為了微小收益引入醜陋的複雜度

**有界模式考量：** 若剩餘迭代次數有限（少於 3 次），優先善用成功而非探索。

## 第 3 階段：修改（一個原子性變更）

- 對範圍內的檔案做**一個**有焦點的變更
- 這個變更應該能用一句話解釋清楚
- 在做變更**之前**先寫下描述（強迫思路清晰）

## 第 4 階段：提交（驗證前）

```bash
git add <changed-files>
git commit -m "experiment: <一句話描述>"
```

在執行驗證**之前** commit，這樣還原時很乾淨：`git reset --hard HEAD~1`

## 第 5 階段：驗證（僅機械化）

執行約定的驗證指令，擷取輸出。

**逾時規則：** 若驗證超過正常時間的 2 倍，強制終止並視為崩潰。

**提取指標：** 解析驗證輸出中的特定指標數值。

## 第 5.5 階段：防護（回歸檢查）

若設定階段定義了**防護（guard）**指令，在驗證後執行。

防護是一個必須**始終通過**的指令——它在優化主要指標的同時保護現有功能。常見防護：`npm test`、`npm run typecheck`、`pytest`、`cargo test`。

**關鍵區別：**
- **驗證（Verify）** 回答：「指標有改善嗎？」（目標）
- **防護（Guard）** 回答：「其他東西有壞掉嗎？」（安全網）

**防護規則：**
- 只在設定了防護時執行（防護是可選的）
- 在驗證之後執行——若指標沒改善，就沒必要檢查防護
- 防護只有通過/失敗（exit code 0 = 通過），不需要提取指標
- 若防護失敗，還原優化方案並嘗試重新設計（最多 2 次）
- **永遠不要修改防護/測試檔案**——始終調整實作
- 明確記錄防護失敗，讓代理學習哪類變更容易造成回歸

**防護失敗恢復（最多 2 次重新設計）：**

當防護失敗但指標有改善時，優化想法可能仍然可行——只需要不破壞行為的不同實作：

1. 還原變更（`git reset --hard HEAD~1`）
2. 讀取防護輸出，了解什麼損壞了（哪些測試、哪些斷言）
3. 重新設計優化方案以避免回歸，例如：
   - 若內聯函式破壞了呼叫者 → 嘗試不同的優化角度
   - 若修改資料結構破壞了序列化 → 保留介面
   - 若重新排列邏輯破壞了邊界情況 → 更精準地加入優化
4. Commit 重新設計的版本，重新執行驗證 + 防護
5. 若兩者都通過 → 保留；若防護再次失敗 → 再試一次，然後放棄

**重要：** 防護/測試檔案是唯讀的。優化方案必須適應測試，而非反過來。若 2 次重新設計後仍無法通過防護，丟棄此方案，繼續嘗試其他想法。

## 第 6 階段：決策（無歧義）

```
IF metric_improved AND (no guard OR guard_passed):
    STATUS = "keep"
    # 不做任何事——commit 保留
ELIF metric_improved AND guard_failed:
    git reset --hard HEAD~1
    # 重新設計優化方案（最多 2 次）
    FOR attempt IN 1..2:
        分析防護輸出 → 重新設計實作（不是測試）
        git add + commit 重新設計版本
        重新執行驗證
        IF metric_improved:
            重新執行防護
            IF guard_passed:
                STATUS = "keep (reworked)"
                BREAK
        git reset --hard HEAD~1
    IF 2 次後仍失敗:
        STATUS = "discard"
        REASON = "guard failed, could not rework optimization"
ELIF metric_same_or_worse:
    STATUS = "discard"
    git reset --hard HEAD~1
ELIF crashed:
    # 嘗試修復（最多 3 次）
    IF fixable:
        Fix → re-commit → re-verify → re-guard
    ELSE:
        STATUS = "crash"
        git reset --hard HEAD~1
```

**簡化覆蓋：** 若指標勉強改善（+<0.1%）但變更增加了顯著複雜度，視為 "discard"。若指標不變但程式碼更簡潔，視為 "keep"。

## 第 7 階段：記錄結果

追加到結果日誌（TSV 格式）：

```
iteration  commit   metric   status   description
42         a1b2c3d  0.9821   keep     增加 attention heads 從 8 到 12
43         -        0.9845   discard  切換優化器到 SGD
44         -        0.0000   crash    加倍 batch size（OOM）
```

## 第 8 階段：重複

### 無界模式（預設）

回到第 1 階段。**永遠不要停止。永遠不要問「要繼續嗎？」**

### 有界模式（搭配 /loop N）

```
IF current_iteration < max_iterations:
    回到第 1 階段
ELIF goal_achieved:
    Print: "目標在迭代 {N} 達成！最終指標：{value}"
    印出最終摘要
    STOP
ELSE:
    印出最終摘要
    STOP
```

**最終摘要格式：**
```
=== Autoresearch 完成（N/N 次迭代）===
基準線：{baseline} → 最終：{current}（{delta}）
保留：X | 丟棄：Y | 崩潰：Z
最佳迭代：#{n} — {description}
```

### 陷入困境時（連續 5 次以上丟棄）

適用於兩種模式：
1. 重新從頭讀取所有範圍內的檔案
2. 重新讀取原始目標/方向
3. 檢視完整結果日誌尋找模式
4. 嘗試組合 2-3 個先前成功的變更
5. 嘗試**與一直失敗的方向相反**的做法
6. 嘗試激進的架構改變

## 崩潰恢復

- 語法錯誤 → 立即修復，不計入獨立迭代
- 執行期錯誤 → 嘗試修復（最多 3 次），然後繼續
- 資源耗盡（OOM）→ 還原，嘗試更小的變體
- 無限迴圈/掛起 → 逾時後終止，還原，避免該方法
- 外部依賴失敗 → 跳過，記錄，嘗試不同方法

## 溝通

- **不要**問「我該繼續嗎？」——無界模式：是，永遠繼續；有界模式：繼續直到達到 N 次
- **不要**在每次迭代後做摘要——只記錄並繼續
- **要**每約 5 次迭代印出一行簡短狀態（例如：「迭代 25：指標 0.95，8 次保留 / 17 次丟棄」）
- **要**在發現令人驚訝或改變局面的事情時提醒
- **要**在有界迴圈完成時印出最終摘要

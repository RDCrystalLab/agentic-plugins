# Excalidraw 圖表建立器

生成 `.excalidraw` JSON 檔案，讓圖表**以視覺來論述**，而非單純陳列資訊。

## 自訂風格

**所有色彩與品牌風格集中在 `references/color-palette.md`。** 生成圖表前先讀取它，作為色彩選擇的唯一真相來源 — 形狀填充、邊框、文字色彩、證據工件背景，一切以此為準。

---

## 核心哲學

**圖表應該論述（argue），而非展示（display）。**

圖表不是格式化文字。它是一個視覺論述，展示文字無法單獨表達的關係、因果與流程。形狀本身就應該是意義。

**同構測試（Isomorphism Test）**：如果移除所有文字，結構本身能傳達概念嗎？如果不能，重新設計。

**教育測試（Education Test）**：看圖的人能從中學到具體的東西嗎？還是它只是在替方框貼標籤？好的圖表會教學 — 展示實際格式、真實的事件名稱、具體範例。

---

## 深度評估（最先做這一步）

在設計之前，判斷圖表需要什麼程度的細節：

### 簡單 / 概念性圖表
使用抽象形狀的時機：
- 解釋思維模型或理念
- 受眾不需要技術細節
- 概念本身就是抽象的（例如「關注點分離」）

### 全面 / 技術性圖表
使用具體範例的時機：
- 繪製真實系統、協定或架構
- 圖表用於教學或解釋
- 受眾需要了解事物的真實樣貌
- 展示多個技術如何整合

**技術性圖表必須包含證據工件**（見下方）。

---

## 研究指令（技術性圖表）

**在繪製任何技術內容之前，先研究實際規格。**

如果要繪製協定、API 或框架：
1. 查詢實際的 JSON / 資料格式
2. 找到真實的事件名稱、方法名稱或 API 端點
3. 了解各部件如何實際連接
4. 使用真實術語，而非泛用佔位符

壞例子："Protocol" -> "Frontend"
好例子："AG-UI streams events (RUN_STARTED, STATE_DELTA)" -> "CopilotKit renders via createA2UIMessageRenderer()"

---

## 證據工件（Evidence Artifacts）

證據工件是具體範例，用以證明圖表的準確性並幫助觀者學習。技術性圖表應包含這些。

| 工件類型 | 使用時機 | 渲染方式 |
|----------|----------|----------|
| **程式碼片段** | API、整合、實作細節 | 深色矩形 + 語法高亮文字（見 color palette） |
| **資料 / JSON 範例** | 資料格式、schema、payload | 深色矩形 + 彩色文字 |
| **事件 / 步驟序列** | 協定、工作流程 | 時間軸模式（線 + 圓點 + 標籤） |
| **UI 模擬** | 展示實際輸出 | 巢狀矩形模擬真實 UI |
| **API / 方法名稱** | 真實的函式呼叫、端點 | 使用文件中的實際名稱，非佔位符 |

**關鍵原則：展示事物的真實樣貌**，而非僅標註名稱。

---

## 多層縮放架構（Multi-Zoom）

全面性圖表同時在多個縮放層級運作。

### 第 1 層：摘要流程
簡化的總覽，一目了然展示完整管線。通常放在圖表的頂部或底部。

### 第 2 層：區段邊界
標記區域，將相關元件分組。建立視覺「房間」。

### 第 3 層：區段內的細節
證據工件、程式碼片段和具體範例。教育價值的核心所在。

**全面性圖表應包含所有三個層級。**

---

## 容器 vs. 浮動文字

**不是每段文字都需要形狀包裹。** 預設使用浮動文字。只在有意義時才加容器。

| 使用容器的時機 | 使用浮動文字的時機 |
|----------------|-------------------|
| 它是區段的焦點 | 它是標籤或描述 |
| 需要與其他元素視覺分組 | 它是輔助細節或 metadata |
| 箭頭需要連接到它 | 它在描述附近的東西 |
| 形狀本身帶有意義 | 字體大小本身就足以建立層級 |
| 它代表系統中一個獨立的「東西」 | 它是區段標題、副標題或註解 |

**容器測試**：對每個有框的元素問「這個改成浮動文字行不行？」如果行，就移除容器。

## 設計流程（生成 JSON 之前必做）

### 步驟 0：評估所需深度
判斷是簡單/概念性，還是全面/技術性。如果是全面性，先做研究。

### 步驟 1：深入理解
對每個概念問：
- 這個概念**做什麼**？（不是它**是什麼**）
- 概念之間有什麼關係？
- 核心的轉換或流程是什麼？
- **讀者需要看到什麼才能理解？**

### 步驟 2：將概念映射到模式
為每個概念找到映射其行為的視覺模式（見下方視覺模式庫）。

### 步驟 3：確保多樣性
多概念圖表：**每個主要概念必須使用不同的視覺模式**。禁止千篇一律的卡片方格。

### 步驟 4：勾勒流程
在生成 JSON 之前，在腦中描繪眼睛如何穿越圖表。應有清晰的視覺故事。

### 步驟 5：生成 JSON
此時才開始建立 Excalidraw 元素。**大型圖表見下方分段建構策略。**

### 步驟 6：渲染與驗證（必要）
生成 JSON 後，必須執行 render-view-fix 迴圈直到圖表正確。見下方**渲染與驗證**段落。

---

## 大型 / 全面性圖表策略

**全面性或技術性圖表必須逐段建構 JSON。** 不要嘗試在一次回應中生成整個檔案 — 會導致截斷和品質下降。

### 逐段工作流程

**階段 1：逐段建構**

1. **建立基礎檔案** — 包含 JSON 外殼（`type`、`version`、`appState`、`files`）和第一段元素。
2. **每次編輯加入一個區段。** 每個區段有自己的專屬步驟。
3. **使用描述性字串 ID**（例如 `"trigger_rect"`、`"arrow_fan_left"`）。
4. **按區段命名 seed**（例如區段 1 使用 100xxx，區段 2 使用 200xxx）。
5. **即時更新跨區段 binding** — 當新區段的箭頭連接到先前區段的元素時，同時更新該元素的 `boundElements`。

**階段 2：整體審查**

所有區段就位後，通讀完整 JSON 並檢查：
- 跨區段箭頭的兩端 binding 正確嗎？
- 整體間距平衡嗎？
- 所有 ID 和 binding 都指向實際存在的元素嗎？

**階段 3：渲染與驗證**

執行渲染與驗證段落中的 render-view-fix 迴圈。

### 禁止事項

- **不要在一次回應中生成整個圖表。** 你會撞到 token 上限並產生截斷的 JSON。
- **不要寫 Python 生成腳本。** 使用描述性 ID 的手工 JSON 更容易維護。

## 視覺模式庫

### 扇出（一對多）
中心元素向多個目標輻射箭頭。用於：來源、根因、中樞。
```
        O
       /
  [] --O
       \
        O
```

### 匯聚（多對一）
多個輸入合併到單一輸出。用於：聚合、漏斗、綜合。

### 樹狀（層級）
父子分支結構，使用連線和浮動文字（不需要方框）。
```
  label
  +-- label
  |   +-- label
  |   +-- label
  +-- label
```
使用 `line` 元素作為主幹/分支，浮動文字作為標籤。

### 時間軸（序列）
線 + 圓點 + 浮動標籤。用於：步驟序列、事件流。
```
  *--- 標籤 1
  |
  *--- 標籤 2
  |
  *--- 標籤 3
```

### 螺旋 / 循環（持續迴圈）
元素依序排列，箭頭回到起點。用於：回饋迴圈、迭代流程。

### 雲狀（抽象狀態）
大小不一的重疊橢圓。用於：上下文、記憶、心智狀態。

### 流水線（轉換）
輸入 -> 處理方塊 -> 輸出，有明確的前後對比。用於：資料轉換、pipeline。

### 並排（比較）
兩個平行結構帶有視覺對比。用於：前後對比、取捨。

### 間隔 / 斷裂（分隔）
區段之間的視覺留白或屏障。用於：階段切換、邊界。

### 線條作為結構
使用線條（type: `line`，非 arrow）作為主要結構元素：
- **時間軸**：垂直/水平線 + 間隔圓點
- **樹狀結構**：主幹 + 分支線 + 浮動文字標籤
- **分隔線**：細虛線分隔區段

## 形狀意義

根據概念的代表意義選擇形狀 — 或完全不使用形狀：

| 概念類型 | 形狀 | 原因 |
|----------|------|------|
| 標籤、描述 | **無**（浮動文字） | 字體大小建立層級 |
| 區段標題 | **無**（浮動文字） | 字體大小/粗細就足夠 |
| 時間軸標記 | 小型 `ellipse`（10-20px） | 視覺錨點，非容器 |
| 起始、觸發、輸入 | `ellipse` | 柔和、起源感 |
| 結束、輸出、結果 | `ellipse` | 完成、目的地 |
| 決策、條件 | `diamond` | 經典決策符號 |
| 流程、動作、步驟 | `rectangle` | 有邊界的動作 |
| 抽象狀態、上下文 | 重疊的 `ellipse` | 模糊、雲狀 |
| 層級節點 | 線條 + 文字（無方框） | 用線條建立結構 |

**原則**：預設不使用容器。只在形狀能傳達意義時才加入。

## 色彩即意義

色彩用於編碼資訊，而非裝飾。所有色彩選擇應來自 `references/color-palette.md`。

**關鍵原則：**
- 每個語意用途有特定的 fill / stroke 配對
- 浮動文字用色彩建立層級（標題、副標題、細節）
- 證據工件使用深色背景 + 彩色文字
- 深色邊框搭配淺色填充以產生對比

**不要自創新色彩。** 若概念不符合現有語意類別，使用 Primary / Neutral。

## 現代美學

### 粗糙度（Roughness）
- `roughness: 0` — 乾淨、俐落的邊緣。**專業圖表的預設值。**
- `roughness: 1` — 手繪、有機感。用於腦力激盪/非正式場合。

### 邊框寬度（Stroke Width）
- `strokeWidth: 1` — 纖細、優雅。線條、分隔線、細微連線。
- `strokeWidth: 2` — 標準。形狀和主要箭頭。
- `strokeWidth: 3` — 粗體。謹慎使用於強調處。

### 不透明度（Opacity）
**所有元素一律使用 `opacity: 100`。** 用色彩、尺寸和邊框寬度建立層級，而非透明度。

### 小型標記
使用小圓點（10-20px ellipse）替代完整形狀，作為時間軸標記、項目符號或視覺錨點。

## 版面原則

### 透過尺寸建立層級
- **主角（Hero）**：300x150 — 視覺錨點，最重要
- **主要（Primary）**：180x90
- **次要（Secondary）**：120x60
- **小型（Small）**：60x40

### 留白 = 重要性
最重要的元素周圍應有最多空白（200px+）。

### 流向
引導視線：序列通常從左到右或從上到下，輻射式用於中樞-輻條（hub-and-spoke）。

### 連線必備
位置本身不能表達關係。如果 A 與 B 有關，必須有箭頭。

### 格線對齊
座標對齊到 10 的倍數（格線為 20px）。

---

## 文字規則

**關鍵**：JSON 中的 `text` 屬性只能包含可讀的文字。

設定：`fontSize: 16`, `fontFamily: 3`, `textAlign: "center"`, `verticalAlign: "middle"`

多行：在 `"text"` 和 `"originalText"` 中使用 `\n`。

## 常見錯誤（必須避免）

### 1) 容器太小、子元素太密集
- 如果矩形內含其他矩形（命名空間/邊界），容器要做得顯著更大。
- 子矩形之間保持可見間距（最小間距：>= 60px）。

### 2) 文字對齊方式與矩形角色不匹配
- **容器矩形**（含其他矩形）：標籤放在左上方（`textAlign: "left"`, `verticalAlign: "top"`）。
- **葉節點矩形**（無子元素）：標籤置中。

### 3) 箭頭被矩形遮蔽
- 確保箭頭在形狀之上渲染（箭頭在 elements 陣列中排在矩形之後）。
- 擁擠區域使用折線路由並增加 `strokeWidth` 到 3。

### 4) 外部長線穿越核心區域
- 長的 ingress/外部箭頭沿外圍邊距路由，不要穿過擁擠的核心區域。

### 5) 中心因標籤和線條而雜亂
- 箭頭標籤保持簡短，定位在彎折處或密集中心之外。
- 減少交叉線比追求最短幾何路徑更重要。

### 6) 千篇一律的卡片方格
- 每個主要概念必須使用不同的視覺模式。禁止等大方框排列。

### 7) 每段文字都加方框
- 預設使用浮動文字。目標：<30% 的文字元素位於容器內。

## 渲染與驗證（必要步驟）

單從 JSON 無法判斷圖表品質。生成或編輯 Excalidraw JSON 後，必須渲染為 PNG、檢視圖片、修正看到的問題。

### 如何渲染

```bash
uv run tools/excalidraw-to-png.py <path-to-file.excalidraw>
```

這會在原始檔案旁輸出 `.excalidraw.png`。然後使用 **Read tool** 檢視 PNG。

### 迴圈流程

**1. 渲染並檢視** — 執行渲染腳本，然後 Read PNG。

**2. 對照原始設計審查：**
- 視覺結構是否符合你規劃的概念結構？
- 每個區段是否使用了預期的模式？
- 視線是否按設計順序流過圖表？
- 視覺層級正確嗎 — 主角元素佔優勢、輔助元素較小？
- 技術性圖表：證據工件是否可讀且正確放置？

**3. 檢查視覺缺陷：**
- 文字被容器裁切或溢出
- 文字或形狀與其他元素重疊
- 箭頭穿越元素而非繞過
- 標籤位置模糊不清
- 間距不均或構圖失衡
- 文字在渲染尺寸下太小難以閱讀

**4. 修正** — 編輯 JSON 解決所有問題。常見修正：
- 擴大容器以容納裁切的文字
- 調整 `x`/`y` 座標改善間距和對齊
- 在箭頭 `points` 陣列中加入中繼點
- 調整元素尺寸以重新平衡視覺重量

**5. 重新渲染並重新檢視** — 重複直到圖表通過設計審查和缺陷檢查。通常需要 2-4 次迭代。

### 首次設定

```bash
uv run --with 'playwright>=1.40.0' python -m playwright install chromium
```

## 品質清單

### 深度與佐證（技術性圖表）
1. **研究完成**：是否查詢了實際規格、格式、事件名稱？
2. **證據工件**：是否包含程式碼片段、JSON 範例或真實資料？
3. **多層縮放**：是否有摘要流程 + 區段邊界 + 細節？
4. **具體優於抽象**：展示真實內容，而非僅有標籤方框？

### 概念性
5. **同構性**：每個視覺結構是否映射其概念的行為？
6. **論述性**：圖表是否展示了文字無法表達的東西？
7. **多樣性**：每個主要概念是否使用不同的視覺模式？
8. **無千篇一律容器**：避免了卡片方格和等大方框？

### 容器紀律
9. **最少容器**：是否有框的元素可以改為浮動文字？
10. **線條作為結構**：樹狀/時間軸模式是否使用線條+文字而非方框？
11. **字體層級**：字體大小和色彩是否建立了視覺層級？

### 結構性
12. **連線**：每個關係都有箭頭或線條
13. **流向**：視線有清晰的路徑可循
14. **層級**：重要元素更大/更孤立

### 技術性
15. **文字乾淨**：`text` 只包含可讀文字
16. **字體**：`fontFamily: 3`
17. **粗糙度**：`roughness: 0`（除非要求手繪風格）
18. **不透明度**：所有元素 `opacity: 100`
19. **ID**：描述性字串 ID，非數字

### 視覺驗證（必須渲染）
20. **已渲染為 PNG**：圖表已被渲染並目視檢查
21. **無文字溢出**：所有文字都在容器內
22. **無重疊元素**：形狀和文字不會意外重疊
23. **間距均勻**：相似元素有一致的間距
24. **箭頭準確落點**：箭頭連接到預期元素
25. **渲染尺寸下可讀**：文字在 PNG 中清晰可辨
26. **構圖平衡**：沒有大面積空洞或過度擁擠的區域

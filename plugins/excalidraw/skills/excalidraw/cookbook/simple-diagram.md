# 簡單 / 概念性圖表

當圖表是概念性的、抽象的，或受眾不需要技術細節時，使用此 cookbook。

## 適用時機

- 解釋思維模型或理念
- 不需要實作細節的高層次概覽
- 概念本身就是抽象的（例如「關注點分離」）

## 設計流程

### 步驟 1：深入理解
對每個概念問：
- 這個概念**做什麼**？（不是它**是什麼**）
- 概念之間有什麼關係？
- 核心的轉換或流程是什麼？
- **讀者需要看到什麼才能理解？**

### 步驟 2：將概念映射到視覺模式
為每個概念找到映射其行為的視覺模式。讀取 `cookbook/visual-patterns.md` 取得模式庫。

### 步驟 3：確保多樣性
多概念圖表：**每個主要概念必須使用不同的視覺模式**。禁止千篇一律的卡片方格。

### 步驟 4：勾勒流程
在生成 JSON 之前，在腦中描繪眼睛如何穿越圖表。應有清晰的視覺故事。

### 步驟 5：生成 JSON
此時才開始建立 Excalidraw 元素。讀取 `references/element-templates.md` 取得 JSON 模板，讀取 `references/color-palette.md` 取得色彩。

### 步驟 6：渲染與驗證
生成 JSON 後，必須執行 render-view-fix 迴圈直到圖表正確（見 SKILL.md 的渲染與驗證段落）。

## 容器紀律

**不是每段文字都需要形狀包裹。** 預設使用浮動文字。只在有意義時才加容器。

| 使用容器的時機 | 使用浮動文字的時機 |
|----------------|-------------------|
| 它是區段的焦點 | 它是標籤或描述 |
| 需要與其他元素視覺分組 | 它是輔助細節或 metadata |
| 箭頭需要連接到它 | 它在描述附近的東西 |
| 形狀本身帶有意義 | 字體大小本身就足以建立層級 |
| 它代表系統中一個獨立的「東西」 | 它是區段標題、副標題或註解 |

**容器測試**：對每個有框的元素問「這個改成浮動文字行不行？」如果行，就移除容器。

## 品質清單

1. **同構性**：每個視覺結構是否映射其概念的行為？
2. **論述性**：圖表是否展示了文字無法表達的東西？
3. **多樣性**：每個主要概念是否使用不同的視覺模式？
4. **無千篇一律容器**：避免了卡片方格和等大方框？
5. **最少容器**：是否有框的元素可以改為浮動文字？
6. **線條作為結構**：樹狀/時間軸模式是否使用線條+文字而非方框？
7. **字體層級**：字體大小和色彩是否建立了視覺層級？
8. **連線完整**：每個關係都有箭頭或線條？
9. **流向清晰**：視線有清晰的路徑可循？
10. **層級分明**：重要元素更大/更孤立？
